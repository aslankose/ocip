## Abstract

OCIP-0003 defines the **Group Session** object — a portable, multi-participant, multi-platform conversation layer. A Group Session enables multiple humans, each using their own AI platform or model, to contribute to a shared conversation thread. Contributions are attributed to individual participants, and the full session context — including which AI assisted each turn and the settled economic cost of each inference operation — is preserved and portable.

OCIP-0003 defines two sync modes:

- **Async mode** — participants exchange session state via OCIP files or links, like an email thread
- **Live sync mode** — participants exchange session state in real time via a defined messaging protocol

Both modes use the same session object. The sync mode determines how the object is transported, not how it is structured.

## 1. Motivation

Real collaborative work on AI-assisted tasks involves multiple people — each with their preferred AI platform, their own model of choice, and their own context. Today there is no way for these people to work in a shared thread across platforms.

This is the gap OCIP-0003 fills. It is the equivalent of RCS for AI-assisted group communication — a universal session layer that works regardless of which AI platform each participant uses.

**The four scenarios OCIP-0003 enables:**

### Scenario 1: Async collaboration

Aslan starts a research thread on Claude. He exports the session and shares it with a colleague who continues it on ChatGPT. The colleague exports the updated session and returns it. The full turn history — including which AI assisted each turn — is preserved throughout.

### Scenario 2: Live multi-platform session

A team of four works simultaneously on a shared AI-assisted document. Each member uses a different AI. All turns appear in a shared real-time thread. Each turn is attributed to its human author and the AI that assisted them.

### Scenario 3: Model routing within a group

Mid-session, one participant determines their model is not performing well for a specific sub-task. They hand off their contribution to a different model using the `handoff` object from OCIP-0005, without disrupting the shared thread.

### Scenario 4: Federated compute collaboration

Two institutions, each running specialized local models on contributed compute via Inferix, collaborate on a shared research problem. Each inference operation is attributed to the querying participant, settled against the Inferix token ledger, and recorded in the session for full economic auditability.

## 2. Design Principles

**1. Participant identity is explicit.** Every turn in a group session MUST be attributed to a human participant. The AI is a tool, not a participant.

**2. Platform diversity is a first-class concern.** The session schema MUST support participants on different platforms and models simultaneously.

**3. Async and live sync use the same object.** The session structure does not change between modes. Only the transport and update mechanism differs.

**4. Turn ordering is authoritative.** The `sequence` field on each turn provides a global ordering across all participants. Conflicts MUST be resolved by sequence number.

**5. Backwards compatibility.** A Group Session that contains turns from a single participant is a valid OCIP-0005 conversation. The session layer adds to, not replaces, the conversation layer.

**6. Live sync is opt-in.** Implementations that support only async mode are fully conformant. Live sync is an extension.

**7. Economic records are settled, not speculative.** The `inference_cost` field on a turn MUST only be written after ledger confirmation. Pending costs MUST NOT appear in session records.

**8. Economic fields are optional by design.** Sessions not connected to Inferix are fully valid OCIP-0003 documents. Implementations MUST ignore `inference_cost` if not Inferix-connected and MUST NOT reject documents that contain it.

## 3. Terminology

| Term | Definition |
|---|---|
| **Group Session** | A multi-participant, multi-platform conversation thread |
| **Participant** | A human contributor to the session, identified by a stable ID |
| **SessionTurn** | A single contribution within a session, attributed to a participant |
| **Sequence number** | A global monotonically increasing integer that orders turns across all participants |
| **Async mode** | Session state exchanged via file or link, like an email thread |
| **Live sync mode** | Session state exchanged in real time via a defined protocol |
| **Session host** | The participant or service responsible for maintaining authoritative session state in live sync mode |
| **Delta** | A partial session update containing only new turns since a given sequence number |
| **GFT** | GigaFLOP-Token — the compute credit unit used by Inferix, denominated in normalized floating-point operations |
| **Ledger confirmation** | A finalized, hash-chained entry in the Inferix token ledger recording a settled inference transaction |

### 5.4 SessionTurn

A single contribution within a group session. Extends the Turn concept from OCIP-0005 with participant attribution, global sequence ordering, optional content type hint, and optional settled inference cost.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string (UUID) | **REQUIRED** | Unique identifier for this turn. |
| `sequence` | integer | **REQUIRED** | Global monotonically increasing sequence number. Determines authoritative turn order across all participants. MUST be unique within a session. |
| `participant_id` | string (UUID) | **REQUIRED** | References a `Participant.id` in this session. Every turn MUST be attributed to a human participant. |
| `role` | enum | **REQUIRED** | MUST be one of: `"user"`, `"assistant"`, `"system"`. Retains OCIP-0005 semantics. |
| `content` | string | **REQUIRED** | The content of this turn. MUST NOT be null or empty. |
| `content_type` | string | OPTIONAL | MIME type hint for the `content` field. Defaults to `"text/plain"` if absent. Common values: `"text/plain"`, `"text/markdown"`, `"multipart/mixed"`. Importers MUST NOT reject a turn with an unrecognized `content_type` — they SHOULD treat it as `"text/plain"` and surface a warning. |
| `timestamp` | string (ISO 8601) | OPTIONAL | When this turn was created. |
| `model` | string | OPTIONAL | Model that generated this turn. Applicable when `role` is `"assistant"`. |
| `platform` | string | OPTIONAL | Platform where this turn was generated. |
| `handoff` | Handoff | OPTIONAL | Present when this turn marks a model or platform switch. See §5.5. |
| `inference_cost` | InferenceCost | OPTIONAL | Settled economic record of the inference operation that produced this turn. MUST NOT be present until Inferix ledger confirmation is received. See §5.4.1. |
| `metadata` | object | OPTIONAL | Free-form extension envelope. |

**Note on `role` in group sessions:** In a group session, `role` describes the nature of the content, not the participant. A `participant_id` always identifies the human who contributed or prompted the turn, even when `role` is `"assistant"` (indicating the AI response generated on behalf of that participant).

**Note on `inference_cost` timing:** A turn MUST be written to the session immediately upon creation. The `inference_cost` field is populated separately, after ledger confirmation is received. Implementations MUST treat a turn without `inference_cost` as either not Inferix-connected or awaiting confirmation — both are valid states.

### 5.4.1 InferenceCost

Records the settled economic cost of a single inference operation within a group session. This object MUST only be written after the Inferix token ledger has confirmed the transaction. Implementations not connected to Inferix MUST ignore this object if present and MUST NOT reject documents that contain it.

**The questioner pays:** the participant who directs a query to another participant's model is the payer. The node that serves the inference earns the corresponding GFT credit. This is recorded explicitly via `payer_participant_id` and `provider_node_id`.

| Field | Type | Required | Description |
|---|---|---|---|
| `gft_amount` | number | **REQUIRED** | GigaFLOP-Tokens debited from the payer's wallet, as confirmed by the Inferix ledger. |
| `flops_delivered` | number | OPTIONAL | Raw FLOPs recorded for this inference operation at the serving node. |
| `payer_participant_id` | string (UUID) | OPTIONAL | The participant whose wallet was debited. If omitted, defaults to the `participant_id` of the turn. MUST be set explicitly when the payer differs from the turn's `participant_id`. |
| `provider_node_id` | string | OPTIONAL | The Inferix node identifier that served this inference. SHOULD match the `inferix_node_id` of the serving participant if they are a session participant. |
| `ledger_ref` | string | **REQUIRED** | Confirmation reference ID from the Inferix token ledger. Used for independent reconciliation between session records and ledger records. |
| `confirmed_at` | string (ISO 8601) | **REQUIRED** | Timestamp of ledger confirmation. MUST reflect ledger time, not local clock or turn creation time. |
| `settlement_status` | enum | **REQUIRED** | MUST be `"confirmed"`. Pending or unconfirmed costs MUST NOT be written to the session record. |

## 6. Sync Modes

### 6.1 Async Mode

In async mode, session state is exchanged as complete OCIP-0003 documents — via file share or link, using the mechanisms defined in OCIP-0005 §7. This is analogous to an email thread.

**Workflow:**

1. Participant A starts a session and contributes turns.
2. Participant A exports the session as an `.ocip3.json` file or link and shares it.
3. Participant B imports the session, adds their turns (with their own `participant_id`), and exports the updated session.
4. The updated session is returned to Participant A, who imports and continues.

**Delta exports:** In async mode, implementations MAY export a **delta** — a partial session containing only turns with `sequence` greater than a specified value. A delta MUST include the full `participants[]` array and session metadata. The recipient MUST merge delta turns into their local session by sequence number, discarding duplicates.

**Conflict resolution:** If two participants add turns at the same sequence number independently, the receiving implementation MUST assign new sequence numbers to the incoming turns, appending them after the local highest sequence.

**Merge algorithm:**

1. Identify the highest `sequence` in the local session — call it `local_max`.
2. For each incoming turn where `sequence` > `local_max`: append as-is.
3. For each incoming turn where `sequence` <= `local_max` and `id` is not present locally: assign `sequence = local_max + offset`, increment offset, add `metadata.merge_note` recording original sequence and merge timestamp.
4. For each incoming turn where `id` is already present locally: discard (duplicate).
5. Re-sort all turns by final `sequence`.

---

### 6.2 Live Sync Mode

In live sync mode, session state is maintained by a **session host** and participants receive and send turn updates in real time. The session object structure is identical to async mode. The difference is in how updates are transported.

When `sync_mode` is `"live"`, the `live_sync` configuration object MUST be present in the root document.

#### LiveSyncConfig Object

| Field | Type | Required | Description |
|---|---|---|---|
| `host_uri` | string | **REQUIRED** | The WebSocket or HTTP endpoint of the session host. |
| `session_token` | string | OPTIONAL | An opaque short-lived token used for participant authentication with the host. Implementations MUST NOT embed long-lived credentials here. |
| `protocol_version` | string | **REQUIRED** | The live sync protocol version. MUST be `"1.0"` for this revision. |
| `heartbeat_interval_ms` | integer | OPTIONAL | Expected heartbeat interval in milliseconds. Default: 30000. |

---

### 6.3 Live Sync Message Protocol

The live sync protocol uses JSON messages over WebSocket. All messages share a common envelope:

```json
{
  "ocip_msg": "1.0",
  "type": "<message_type>",
  "session_id": "<UUID>",
  "timestamp": "<ISO 8601>",
  "payload": { }
}
```

#### Message Types

| Type | Direction | Description |
|---|---|---|
| `join` | Client → Host | Participant joining the session. Payload: `{ "participant": Participant }` |
| `joined` | Host → Client | Acknowledgment. Payload: `{ "session": Session, "sequence_at": integer }` |
| `turn` | Client → Host | New turn submitted by a participant. Payload: `{ "turn": SessionTurn (without sequence) }` |
| `turn_ack` | Host → Client | Turn accepted and assigned sequence number. Payload: `{ "turn_id": string, "sequence": integer }` |
| `turn_broadcast` | Host → All | New turn broadcast to all participants. Payload: `{ "turn": SessionTurn (with sequence) }` |
| `cost_update` | Host → All | Inference cost settled and attached to a turn. Payload: `{ "turn_id": string, "inference_cost": InferenceCost }` |
| `delta` | Host → Client | Catch-up delta for a participant rejoining. Payload: `{ "turns": SessionTurn[], "from_sequence": integer }` |
| `heartbeat` | Host → Client | Keepalive. Payload: `{}` |
| `heartbeat_ack` | Client → Host | Keepalive acknowledgment. Payload: `{}` |
| `leave` | Client → Host | Participant leaving the session. Payload: `{ "participant_id": string }` |
| `error` | Host → Client | Error notification. Payload: `{ "code": string, "message": string }` |

**Note on `cost_update`:** Because `inference_cost` is only written after ledger confirmation, it arrives asynchronously after the turn. The `cost_update` message allows the host to push a confirmed cost to all participants and attach it to the correct turn by `turn_id`. Clients MUST apply `cost_update` payloads to the matching turn in their local session state.

#### Sequence Authority

The session host is the authoritative source of sequence numbers in live sync mode. Clients MUST NOT assign sequence numbers — they submit turns without a sequence and receive the assigned sequence in `turn_ack`. The host MUST assign sequence numbers monotonically and broadcast each turn to all connected participants via `turn_broadcast`.

#### Turn Submission and Acknowledgment

Clients MUST include a client-generated `turn.id` (UUID) in each submitted turn so they can correlate incoming `turn_ack` messages. If a client submits a turn and does not receive a `turn_ack` within a recommended window of 10 seconds, it SHOULD resubmit the turn with the same `turn.id`. The host MUST deduplicate resubmitted turns by `turn.id`.

#### Reconnection

If a participant disconnects and reconnects, they MUST send a `join` message. The host SHOULD respond with a `delta` containing all turns since the participant's last known sequence. The participant MUST apply the delta before resuming contribution.

## 7. File Format

- OCIP-0003 Documents MUST be encoded as UTF-8.
- OCIP-0003 Documents MUST be valid JSON per RFC 8259.
- Recommended file extension: `.ocip3.json` or `.ocip.json`
- Recommended MIME type: `application/ocip+json`
- The `ocip_spec` field MUST be `"OCIP-0003"` to distinguish from other OCIP documents.

## 8. Conformance

### Async Conformance (baseline)

A conforming async implementation MUST:

- Produce and consume valid OCIP-0003 documents with `sync_mode: "async"`
- Assign unique sequence numbers to all turns
- Preserve participant attribution on all turns
- Handle delta imports by merging on sequence number using the algorithm defined in §6.1
- Ignore `inference_cost` fields if not Inferix-connected
- NOT reject documents containing `inference_cost`
- Treat a turn with an unrecognized `content_type` as `"text/plain"` and surface a warning rather than rejecting the document

### Live Sync Conformance (extension)

A conforming live sync implementation MUST:

- Implement the LiveSyncConfig object
- Implement all message types defined in §6.3, including `cost_update`
- Maintain sequence number authority at the host
- Implement delta catch-up on participant reconnection
- Implement turn deduplication by `turn.id` on resubmission

A conforming live sync implementation SHOULD:

- Implement heartbeat/heartbeat_ack
- Document maximum session size and participant limits

### Inferix-Connected Conformance (extension)

A conforming Inferix-connected implementation MUST:

- NOT populate `inference_cost` before ledger confirmation
- Use `ledger_ref` from the confirmed Inferix ledger transaction
- Record `confirmed_at` from ledger timestamp, not local clock
- Set `payer_participant_id` explicitly when the payer differs from the turn's `participant_id`
- Emit `cost_update` messages in live sync mode upon ledger confirmation

### Governed Session Conformance (extension)

A conforming governed session implementation MUST:

- Require a valid, non-expired OCIP-0002 `CollaborationAgreement` before admitting participants to the session
- Include the `agreement_id` in `session.metadata`
- Reject participants whose OCIP-0002 declaration version has changed since the agreement was signed without renewal

Implementations that support only async mode are fully conformant at the baseline level and SHOULD declare this in their documentation.

## 9. Privacy and Security Considerations

### Participant Identity

`display_name` and `participant_id` are user-controlled. Implementations MUST NOT assume participant identity is verified unless an external authentication layer such as OCIP-0002 is in use.

### Live Sync Tokens

`session_token` in `LiveSyncConfig` MUST be short-lived. Implementations MUST NOT store long-lived credentials in OCIP documents. Tokens SHOULD expire no later than the session's `share.expires_at`.

### Turn Attribution

Implementations SHOULD allow participants to review and redact their own turns before export. A participant MUST NOT be able to modify or delete turns attributed to another participant.

### Multi-Party Privacy

In a group session, all participants can see all turns by design. Users SHOULD be informed of this before joining a session with `scope: "link"` or `scope: "public"`.

### Inference Cost Records

`inference_cost` records are economic audit trails. Implementations MUST NOT allow modification of a confirmed `inference_cost` field after it has been written. The `ledger_ref` provides an independent verification path against the Inferix ledger.

### No Credentials

Implementations MUST NOT store API keys, platform passwords, long-lived tokens, or Inferix wallet credentials in any OCIP-0003 field.

## 10. Versioning

OCIP-0003 follows the same versioning policy as OCIP-0005. See OCIP-0005 §10.

The live sync protocol has its own version field (`protocol_version` in `LiveSyncConfig`). Changes to the live sync protocol that are not backwards compatible MUST increment the `protocol_version`.

## 11. Relationship to OCIP-0005, OCIP-0004, and OCIP-0002

OCIP-0003 builds on all prior specs:

- The `SessionTurn` object extends `Turn` from OCIP-0005 with `participant_id`, `sequence`, `content_type`, and `inference_cost`.
- The `Handoff` and `ContextSnapshot` objects are reused from OCIP-0005.
- A Group Session MAY be associated with a Project (OCIP-0004) by including `{ "project_id": "<UUID>" }` in `session.metadata`.
- A Group Session established under an OCIP-0002 collaboration agreement SHOULD include `{ "agreement_id": "<UUID>" }` in `session.metadata`. Governed sessions MUST include a valid, non-expired `agreement_id`.
- A Group Session with a single participant and no `live_sync` config is functionally equivalent to an OCIP-0005 conversation.
- The `InferenceCost` object bridges OCIP-0003 sessions with the Inferix token ledger, enabling economic auditability of federated compute collaborations.
- OCIP-0002 participant `node_id` values MAY be linked to OCIP-0003 `participant_id` values via participant `metadata.ocip4_node_id`.
