## Abstract

OCIP-0004 defines the **Project** object — a portable container for multi-conversation AI work. A Project groups related conversations, system instructions, and file attachments into a single transferable unit. Like OCIP-0005, a Project object is platform-agnostic and transport-agnostic. It may be shared between users, moved across AI platforms, or archived as a self-contained artifact.

## 1. Motivation

AI platforms increasingly support persistent "projects" — named workspaces that group conversations with shared system instructions, uploaded files, and custom settings. These projects have no portable representation. When a user wants to:

- Move a project from one AI platform to another
- Share a project with a colleague on a different platform
- Archive a project as a durable, self-contained artifact
- Hand off a project mid-work to a different AI model

...there is no standard format to do so. OCIP-0004 fills this gap.

A Project is a higher-order container. It does not replace the conversation object defined in OCIP-0005 — it references it. A conversation can exist independently (OCIP-0005) or as part of a project (OCIP-0004).

---

## 2. Design Principles

**1. Conversations are first-class.** A Project references OCIP-0005 conversation objects. It does not redefine the conversation schema.

**2. Files travel with intent.** The attachment model is tiered — small files may be embedded inline, large files are referenced with a checksum. The threshold is implementation-defined. The spec defines the envelope, not the size limit.

**3. Instructions are portable.** System prompts and custom instructions are a core part of project context. They MUST be included in a portable project object.

**4. Graceful degradation.** An importer that cannot process attachments MUST still be able to import conversations and instructions. Missing attachments SHOULD surface a warning, not a failure.

**5. No vendor lock-in.** Project objects MUST NOT assume any platform-specific feature set.

---

## 3. Terminology

| Term | Definition |
|---|---|
| **Project** | A named container grouping conversations, instructions, and attachments |
| **Instructions** | System-level prompts or behavioral rules that apply across all conversations in the project |
| **InstructionBlock** | A single labelled instruction unit within a multi-block instruction set |
| **Attachment** | A file associated with the project, carried inline or by reference |
| **Inline attachment** | An attachment whose content is base64-encoded within the document |
| **Reference attachment** | An attachment identified by URI, with a checksum for integrity verification |
| **ConversationRef** | A reference to an OCIP-0005 conversation, either embedded or external |
| **Checksum** | A SHA-256 hash of the attachment content, used for integrity verification |

## 4. Schema Overview

```
OCIPProject
├── ocip_version              (string, required)
├── ocip_spec                 (string, required — "OCIP-0004")
├── doc_id                    (UUID, required)
├── created_at                (ISO 8601, required)
├── project                   (object, required)
│   ├── id                    (UUID, required)
│   ├── title                 (string, required)
│   ├── description           (string, optional)
│   ├── created_at            (ISO 8601, optional)
│   ├── updated_at            (ISO 8601, optional)
│   ├── instructions          (Instructions, optional)
│   ├── attachments[]         (array of Attachment, optional)
│   ├── conversations[]       (array of ConversationRef, optional)
│   └── metadata              (object, optional)
└── share                     (ShareEnvelope, optional)
```

## 5. Object Definitions

### 5.1 OCIPProject

The root object of every OCIP-0004 file.

| Field | Type | Required | Description |
|---|---|---|---|
| `ocip_version` | string | **REQUIRED** | MUST be `"0.1"` for this revision. |
| `ocip_spec` | string | **REQUIRED** | MUST be `"OCIP-0004"`. Identifies this as a Project document. |
| `doc_id` | string (UUID) | **REQUIRED** | Unique identifier for this document instance. |
| `created_at` | string (ISO 8601) | **REQUIRED** | Timestamp when this document was generated. |
| `project` | Project | **REQUIRED** | The project object. See §5.2. |
| `share` | ShareEnvelope | OPTIONAL | Present when the document is intended for sharing. See §5.7. |

---

### 5.2 Project

The core project container.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string (UUID) | **REQUIRED** | Stable unique identifier for this project. |
| `title` | string | **REQUIRED** | Human-readable project name. |
| `description` | string | OPTIONAL | A brief description of the project's purpose. |
| `created_at` | string (ISO 8601) | OPTIONAL | When the project was created. |
| `updated_at` | string (ISO 8601) | OPTIONAL | When the project was last modified. |
| `instructions` | Instructions | OPTIONAL | System-level instructions for this project. See §5.3. |
| `attachments` | array of Attachment | OPTIONAL | Files associated with this project. See §5.4. |
| `conversations` | array of ConversationRef | OPTIONAL | Conversations belonging to this project. See §5.6. |
| `metadata` | object | OPTIONAL | Free-form extension envelope. Unknown keys MUST be ignored. Projects established under an OCIP-0002 collaboration agreement SHOULD include `{ "agreement_id": "<UUID>" }` here to record the governance basis. |

### 5.3 Instructions

System-level prompts or behavioral rules that apply to all conversations within the project. This is the portable equivalent of a platform's "custom instructions" or "system prompt" feature.

The `instructions` field accepts either a **single block** (string shorthand) or an **array of InstructionBlocks** for layered prompts. Both forms are valid. Importers MUST support both.

#### Single-block form (string shorthand)

```json
{
  "instructions": {
    "content": "You are a senior software architect. Be precise and cite trade-offs explicitly.",
    "label": "Persona"
  }
}
```

#### Multi-block form (array of InstructionBlocks)

```json
{
  "instructions": {
    "blocks": [
      { "label": "Persona", "content": "You are a senior software architect." },
      { "label": "Constraints", "content": "Always cite trade-offs. Avoid generic advice." },
      { "label": "Domain context", "content": "The system processes 50k financial transactions per second." }
    ]
  }
}
```

**Instructions object fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `content` | string | CONDITIONAL | The instruction text for single-block form. REQUIRED when `blocks` is absent. MAY be multi-line. |
| `blocks` | array of InstructionBlock | CONDITIONAL | Array of labelled instruction units for multi-block form. REQUIRED when `content` is absent. MUST contain at least one block. |
| `role` | enum | OPTIONAL | MUST be `"system"` if present. Default: `"system"`. |
| `label` | string | OPTIONAL | A human-readable label for this instruction set when using single-block form. |
| `metadata` | object | OPTIONAL | Free-form extension envelope. |

An `Instructions` object MUST include either `content` or `blocks`. If both are present, `blocks` takes precedence.

**InstructionBlock fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `label` | string | **REQUIRED** | Human-readable label for this block (e.g. `"Persona"`, `"Constraints"`, `"Domain context"`). |
| `content` | string | **REQUIRED** | The instruction text for this block. MAY be multi-line. |

**Privacy note:** Instructions MAY contain sensitive organizational or personal information. Exporters MUST warn users when instructions are present and allow their exclusion before export or share.

### 5.4 Attachment

Represents a file associated with the project. The `embed_strategy` field determines whether the content is carried inline or by reference.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string (UUID) | **REQUIRED** | Unique identifier for this attachment within the project. |
| `filename` | string | **REQUIRED** | Original filename including extension (e.g. `"requirements.pdf"`). |
| `mime_type` | string | **REQUIRED** | MIME type of the file (e.g. `"application/pdf"`, `"text/plain"`). |
| `size_bytes` | integer | OPTIONAL | File size in bytes at time of export. |
| `embed_strategy` | enum | **REQUIRED** | MUST be one of: `"inline"`, `"reference"`. See §6. |
| `content` | AttachmentContent | OPTIONAL | Present only when `embed_strategy` is `"inline"`. See §5.5. |
| `uri` | string | OPTIONAL | Present only when `embed_strategy` is `"reference"`. A URI where the file can be retrieved. |
| `checksum` | string | **REQUIRED** | SHA-256 hash of the file content. MUST always be present regardless of embed strategy. Used for integrity verification. Format: `"sha256:<hex-digest>"`. |
| `description` | string | OPTIONAL | A human-readable description of the file's role in the project. |
| `metadata` | object | OPTIONAL | Free-form extension envelope. |

---

### 5.5 AttachmentContent

The inline content of an attachment. Present only when `embed_strategy` is `"inline"`.

| Field | Type | Required | Description |
|---|---|---|---|
| `encoding` | enum | **REQUIRED** | MUST be `"base64"`. |
| `data` | string | **REQUIRED** | The base64-encoded file content. |

---

### 5.6 ConversationRef

A reference to an OCIP-0005 conversation that belongs to this project. Conversations MAY be embedded directly or referenced by ID for external retrieval.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string (UUID) | **REQUIRED** | The conversation ID. MUST match the `id` field of the referenced OCIP-0005 conversation. |
| `title` | string | OPTIONAL | Human-readable title. SHOULD match the title in the referenced conversation. |
| `embed` | object | OPTIONAL | If present, the full OCIP-0005 conversation object is embedded here. The schema MUST conform to OCIP-0005 §5.2. |
| `uri` | string | OPTIONAL | A URI where the full conversation can be retrieved, if not embedded. |
| `order` | integer | OPTIONAL | Display order within the project (1-indexed). |
| `metadata` | object | OPTIONAL | Free-form extension envelope. |

A `ConversationRef` MUST include either `embed` or `uri`. If both are present, `embed` takes precedence. **Importers MUST reject a `ConversationRef` where both `embed` and `uri` are absent** and SHOULD surface an error identifying the affected `id`.

---

### 5.7 ShareEnvelope

Identical to the ShareEnvelope defined in OCIP-0005 §5.6. Reproduced here for self-containment.

| Field | Type | Required | Description |
|---|---|---|---|
| `share_id` | string (UUID) | **REQUIRED** | Unique identifier for this share instance. |
| `created_at` | string (ISO 8601) | **REQUIRED** | When this share was generated. |
| `expires_at` | string (ISO 8601) | OPTIONAL | When this share should be considered expired. |
| `scope` | enum | OPTIONAL | One of: `"private"`, `"link"`, `"public"`. Default: `"private"`. |
| `sender` | string | OPTIONAL | Display name or identifier of the person sharing. |
| `note` | string | OPTIONAL | A human-readable message from sender to recipient. |

## 6. Tiered Attachment Model

OCIP-0004 uses a two-tier model for file attachments. The threshold between tiers is **implementation-defined** — the spec does not prescribe a size limit. Implementations SHOULD document their threshold.

### Tier 1 — Inline

The file content is base64-encoded and embedded directly in the document. Suitable for files where the implementation's threshold permits.

```json
{
  "id": "att-001",
  "filename": "brief.txt",
  "mime_type": "text/plain",
  "size_bytes": 1024,
  "embed_strategy": "inline",
  "checksum": "sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
  "content": {
    "encoding": "base64",
    "data": "SGVsbG8gV29ybGQ="
  }
}
```

### Tier 2 — Reference

The file is too large for inline embedding, or the implementation chooses not to embed it. The URI points to where the file can be retrieved. The checksum allows the recipient to verify integrity after retrieval.

```json
{
  "id": "att-002",
  "filename": "dataset.csv",
  "mime_type": "text/csv",
  "size_bytes": 52428800,
  "embed_strategy": "reference",
  "uri": "https://storage.example.com/files/dataset.csv",
  "checksum": "sha256:abc123def456...",
  "description": "Training dataset used in session 3"
}
```

### Checksum Requirement

The `checksum` field MUST always be present regardless of embed strategy. For inline attachments, the checksum allows importers to verify the embedded content was not corrupted. For reference attachments, it allows the recipient to verify the retrieved file matches what was exported.

Checksum format: `"sha256:<hex-digest>"`.

### Importer Behavior

- If `embed_strategy` is `"inline"`, the importer MUST use the embedded content.
- If `embed_strategy` is `"reference"`, the importer SHOULD attempt to retrieve the file from `uri`. If retrieval fails, the importer MUST surface a warning identifying the missing file by `filename` and MUST NOT fail the entire import.
- Importers SHOULD verify checksums after retrieval and warn the user on mismatch.

## 7. File Format

- OCIP-0004 Documents MUST be encoded as UTF-8.
- OCIP-0004 Documents MUST be valid JSON per RFC 8259.
- Recommended file extension: `.ocip2.json` or `.ocip.json`
- Recommended MIME type: `application/ocip+json`
- The `ocip_spec` field MUST be `"OCIP-0004"` to distinguish from OCIP-0005 and OCIP-0003 documents.

## 8. Conformance

### Exporter Conformance

A conforming Exporter MUST:
- Produce documents that include all REQUIRED fields
- Use ISO 8601 for all timestamps
- Include a `checksum` on every attachment
- Assign a unique UUID to `doc_id`, `project.id`, and each `attachment.id`
- Ensure every `ConversationRef` includes either `embed` or `uri`

A conforming Exporter SHOULD:
- Embed conversations inline using OCIP-0005 schema
- Warn users before exporting instructions
- Document the inline attachment threshold used
- Include `agreement_id` in `project.metadata` when the project was established under an OCIP-0002 collaboration agreement

### Importer Conformance

A conforming Importer MUST:
- Accept any valid OCIP-0004 document
- Ignore unknown fields rather than rejecting the document
- Surface a warning (not an error) for any reference attachment that cannot be retrieved
- Verify checksums on inline attachments
- Reject a `ConversationRef` where both `embed` and `uri` are absent and surface an error identifying the affected conversation `id`
- Support both single-block and multi-block `Instructions` forms

A conforming Importer SHOULD:
- Make imported conversations resumable in the target platform
- Apply imported instructions as system context, preserving block labels where possible
- Present attachment descriptions to the user

## 9. Privacy and Security Considerations

### Instructions

System instructions often contain sensitive organizational context, persona definitions, or behavioral constraints. Exporters MUST warn users when instructions are present and MUST allow their exclusion before export.

### Attachments

Files may contain personally identifiable information, proprietary data, or confidential material. Exporters SHOULD provide a file-by-file review before export.

### Reference URIs

Reference attachment URIs may expose internal storage systems or require authentication. Exporters SHOULD prefer pre-authenticated, time-limited URIs when using `embed_strategy: "reference"`.

### No Credentials

Implementations MUST NOT store API keys, tokens, passwords, or credentials in any OCIP-0004 field, including attachment URIs.

### Checksum Integrity

Importers SHOULD treat a checksum mismatch as a security warning and prompt the user before using the file.

## 11. Relationship to OCIP-0005 and OCIP-0002

OCIP-0004 extends OCIP-0005. The conversation schema defined in OCIP-0005 is used as-is within the `ConversationRef.embed` field. Implementations supporting OCIP-0004 MUST also support OCIP-0005.

A standalone conversation (no project context) uses OCIP-0005. A conversation that is part of a project uses OCIP-0004, with the conversation embedded or referenced within a `ConversationRef`.

OCIP-0004 is governance-aware via OCIP-0002. When a project is established between parties who have completed an OCIP-0002 capability handshake, the resulting `CollaborationAgreement.agreement_id` SHOULD be recorded in `project.metadata` to provide an auditable link between the project and its governance basis. This is RECOMMENDED but not REQUIRED — projects without a governance agreement are fully valid OCIP-0004 documents.
