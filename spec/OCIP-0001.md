## Abstract

OCIP-0001 defines the **Distributed Inference Protocol (DIP)** — a vendor-neutral, open standard for coordinating transformer model inference across a network of volunteer contributor nodes. The protocol covers the full lifecycle of distributed inference: node registration, hardware classification, shard declaration, pipeline discovery, activation passing, reliability verification, token issuance, and fallback routing.

OCIP-0001 is the compute layer standard that underpins the Inferix framework. It is designed to be implementable by any decentralized compute network, not just Inferix. Any system that implements OCIP-0001 can federate with any other conforming implementation, enabling a global cooperative compute layer beneath the session and governance standards defined in OCIP-0002 through OCIP-0005.

**Two-layer design:**

- **Coordination protocol** — standardized HTTP/JSON. How nodes register, heartbeat, report, and receive tasks from the coordination layer. Every implementation MUST speak this.
- **Activation protocol** — transport-agnostic message contract. Defines the schemas for tokenize, forward pass, and activation response. Implementations declare supported transports. The pipeline executor selects based on mutual capability.

## 1. Motivation

Decentralized AI inference networks share a common problem: every implementation defines its own proprietary protocol. A node built for one network cannot participate in another. A client written for one coordinator cannot talk to a different coordinator. Federated compute — where two independent networks pool capacity — is impossible without custom integration work for every pair of systems.

This is the same problem that plagued email before SMTP, calendar before iCalendar, and AI conversations before OCIP (this spec family). The solution is the same: an open, vendor-neutral protocol that any conforming implementation can speak.

OCIP-0001 fills this gap for distributed AI inference. It defines the standard language that:

- A contributor node uses to register its hardware and declare its serving capacity
- A coordination layer uses to discover pipelines, dispatch challenges, and issue tokens
- Shard nodes use to pass activations between themselves during inference
- Any client uses to request inference and redeem tokens

A network that implements OCIP-0001 can federate with any other conforming network. A node built to OCIP-0001 can contribute to any conforming coordinator. This is the foundation for a truly open, global cooperative compute layer.

## 2. Design Principles

**1. Transport agnosticism for activation passing.** The coordination protocol uses HTTP/JSON. The activation passing protocol between shard nodes defines a message contract but leaves transport to the implementation. Nodes declare supported transports. High-performance implementations may use gRPC, WebSocket, QUIC, or any other transport without breaking conformance.

**2. Hardware agnosticism.** The protocol accommodates any hardware from mobile edge devices to data center accelerators. No hardware class is excluded. Token issuance normalizes across hardware via a classification multiplier.

**3. Open participation.** Any node that can pass a benchmark and serve transformer layers can participate. There is no whitelist, no approval process, and no minimum hardware requirement beyond what the model itself demands.

**4. Verifiability.** Compute contributions MUST be verifiable. The reliability challenge protocol provides a mechanism to detect nodes that claim to serve inference without actually doing so. The ledger provides an auditable record of all token issuance and redemption.

**5. Graceful degradation.** When the volunteer pipeline cannot serve a request, the system MUST fall back to a centralized inference service. Users always have a path to redeem tokens. Fallback is transparent to the token economy.

**6. Separation of policy and protocol.** Hardware multiplier values, reliability thresholds, challenge rates, and token expiry periods are policy decisions. The protocol defines how they are applied, not what the values are. Implementations declare their policy parameters.

**7. Federated coordination.** The protocol does not assume a single coordination layer. Multiple conforming coordinators MAY operate independently or in federation. Node identifiers are globally unique UUIDs that do not encode coordinator affiliation.

## 3. Terminology

| Term | Definition |
|---|---|
| **Coordinator** | The service responsible for node registry, pipeline discovery, challenge dispatch, and token ledger |
| **Contributor node** | A participant that serves transformer model layers and earns GFT tokens |
| **Shard** | A contiguous range of transformer layers served by a single contributor node for a specific model |
| **Pipeline** | An ordered set of shards whose layer ranges collectively cover all layers of a model with no gaps |
| **Activation** | The intermediate tensor output of a transformer layer range, passed between shard nodes |
| **Tokenize** | The operation of converting a text prompt into input token IDs, performed by the first shard node |
| **Forward pass** | The operation of processing input through a shard's layer range and producing activations or output tokens |
| **Hardware multiplier** | A scalar that normalizes token issuance across hardware classes, derived from benchmark scores |
| **Reliability factor** | A scalar in [0.5, 1.0] reflecting a node's historical correctness and completion rate |
| **GFT** | GigaFLOP-Token — the compute credit unit, denominated in normalized floating-point operations |
| **Challenge** | A known-prompt inference task dispatched to a node to verify it is serving correct activations |
| **Ledger** | An append-only, hash-chained log of all GFT issuance and redemption events |
| **Fallback** | A centralized inference service used when the volunteer pipeline cannot serve a request |
| **Transport** | The network protocol used for activation passing between shard nodes (e.g. HTTP, gRPC, WebSocket) |

## 4. Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│                      Coordination Layer                      │
│                                                             │
│  Node Registry  │  Pipeline Scheduler  │  Token Ledger     │
│  Challenge Mgr  │  Reliability Tracker │  Fallback Router  │
└────────────┬────────────────┬──────────────────────────────┘
             │                │
     Coordination Protocol    │
     (HTTP/JSON — mandatory)  │
             │                │
    ┌────────▼───────┐        │
    │  Shard Node A  │        │
    │  Layers 0-15   │        │
    │                │        │
    │ Activation     │        │
    │ Server         │        │
    └────────┬───────┘        │
             │                │
     Activation Protocol      │
     (transport-agnostic)     │
             │                │
    ┌────────▼───────┐        │
    │  Shard Node B  │        │
    │  Layers 16-31  │        │
    │                │        │
    │ Activation     │        │
    │ Server         │        │
    └────────┬───────┘        │
             │                │
         Output tokens        │
             │                │
    ┌────────▼───────────────▼┐
    │     Inference Client    │
    │  (token holder)         │
    └─────────────────────────┘
```

**Four principal components:**

**Coordination Layer** — the central orchestration service. Maintains the node registry, discovers pipelines, dispatches reliability challenges, maintains the token ledger, and routes inference requests. The coordination layer exposes the coordination protocol API defined in §13. There MAY be multiple coordination layers operating in federation.

**Contributor Nodes** — participant machines that load transformer model layers and serve inference. Each node registers with the coordination layer, declares its shard assignments, runs an activation server, and responds to periodic reliability challenges. Nodes earn GFT tokens proportional to verified FLOPs delivered.

**Activation Servers** — the inference-serving component running on each contributor node. Accepts tokenize and forward pass requests from the pipeline executor. Passes activations to the next node in the pipeline. The activation server protocol is defined in §9. Transport is implementation-defined.

**Inference Client** — any token-holding participant that submits inference requests. The client interacts with the coordination layer via the coordination protocol. It does not interact directly with shard nodes.

## 5. Object Definitions

### 5.1 NodeRegistration

Submitted by a contributor node to register with the coordination layer.

| Field | Type | Required | Description |
|---|---|---|---|
| `node_id` | string (UUID) | OPTIONAL | If present, MUST be a globally unique UUID. If absent, the coordinator assigns one and returns it. Nodes SHOULD generate their own UUID at first registration and reuse it on reconnection. |
| `public_key` | string | **REQUIRED** | PEM-encoded public key for request authentication. |
| `hardware_profile` | HardwareProfile | **REQUIRED** | Hardware description and benchmark scores. See §5.2. |
| `availability_start` | string (HH:MM UTC) | OPTIONAL | Start of the node's preferred availability window. |
| `availability_end` | string (HH:MM UTC) | OPTIONAL | End of the node's preferred availability window. |
| `max_gpu_utilization` | number | OPTIONAL | Maximum GPU utilization this node will allow for inference. Range [0.0, 1.0]. Default: 0.8. |
| `protocol_version` | string | **REQUIRED** | The OCIP-0001 version this node implements. MUST be `"0.1"` for this revision. |
| `ocip4_node_id` | string (UUID) | OPTIONAL | The node's OCIP-0002 `NodeIdentity.node_id` if it has published a discovery record. Links compute identity to governance identity. |

### 5.2 HardwareProfile

Describes the physical hardware of a contributor node. Benchmark scores are populated by the node's self-benchmark routine and submitted at registration.

| Field | Type | Required | Description |
|---|---|---|---|
| `gpu_model` | string | OPTIONAL | GPU model name (e.g. `"NVIDIA RTX 4090"`). |
| `gpu_memory_gb` | number | OPTIONAL | GPU VRAM in gigabytes. |
| `gpu_type` | enum | OPTIONAL | One of: `"discrete"`, `"integrated"`, `"apple_silicon"`. |
| `cpu_model` | string | OPTIONAL | CPU model name. |
| `cpu_cores` | integer | OPTIONAL | Number of CPU cores. |
| `ram_gb` | number | OPTIONAL | System RAM in gigabytes. |
| `matmul_score` | number | OPTIONAL | Normalized matrix multiplication benchmark score. Range [0.0, 1.0]. |
| `memory_score` | number | OPTIONAL | Normalized memory bandwidth benchmark score. Range [0.0, 1.0]. |
| `latency_score` | number | OPTIONAL | Normalized inference latency benchmark score. Range [0.0, 1.0]. |
| `fingerprint` | string | OPTIONAL | A hardware fingerprint hash derived from device identifiers. Used for Sybil detection. |

**AI Performance Index (API score):**

```
API = 0.5 × matmul_score + 0.3 × memory_score + 0.2 × latency_score
```

If any benchmark score is absent, the coordinator MUST treat the API score as 0.0 and assign the lowest hardware class.

**Hardware penalty:** Nodes with `gpu_type: "integrated"` MUST apply a penalty factor of 0.25 to all benchmark scores before submission. Nodes with no GPU (CPU-only) MUST apply a penalty factor of 0.15. This prevents overestimation of AI compute capability on non-specialist hardware.

### 5.10 ForwardRequest

Sent to each shard node to execute a forward pass through its assigned layer range.

| Field | Type | Required | Description |
|---|---|---|---|

### 5.15 LedgerEntry

A single entry in the coordinator's append-only token ledger. Covers both issuance and redemption events.

| Field | Type | Required | Description |
|---|---|---|---|
| `entry_id` | string (UUID) | **REQUIRED** | Unique identifier for this ledger entry. |
| `event_type` | enum | **REQUIRED** | One of: `"issuance"`, `"redemption"`, `"purchase"`, `"expiry"`, `"penalty"`. |
| `node_id` | string (UUID) | **REQUIRED** | The node whose balance is affected. |
| `amount` | number | **REQUIRED** | GFT amount. Positive for credits, negative for debits. |
| `flops_delivered` | number | OPTIONAL | Raw FLOPs recorded for this operation. Present on `issuance` events. |
| `hardware_multiplier` | number | OPTIONAL | The hardware multiplier applied. Present on `issuance` events. |
| `reliability_factor` | number | OPTIONAL | The reliability factor applied. Present on `issuance` events. |
| `task_id` | string (UUID) | OPTIONAL | The task or inference request that triggered this entry. |
| `inference_request_id` | string (UUID) | OPTIONAL | The inference request ID. Present on `redemption` events. |
| `previous_hash` | string | OPTIONAL | SHA-256 hash of the immediately preceding ledger entry. NULL for the first entry. |
| `entry_hash` | string | **REQUIRED** | SHA-256 hash of this entry's canonical content. Used for chain integrity verification. |
| `timestamp` | string (ISO 8601) | **REQUIRED** | When this entry was recorded. MUST be coordinator time, not node local time. |
| `expires_at` | string (ISO 8601) | OPTIONAL | When the tokens from this issuance entry expire. Present on `issuance` events. |

**Hash content:** The `entry_hash` MUST be computed as SHA-256 of the JSON-serialized object containing: `entry_id`, `event_type`, `node_id`, `amount`, `task_id`, `timestamp`, `previous_hash`. Keys MUST be sorted alphabetically. This ensures deterministic hashing across implementations.

### 5.16 TokenBalance

| Field | Type | Required | Description |
|---|---|---|---|
| `node_id` | string (UUID) | **REQUIRED** | The node identifier. |
| `balance` | number | **REQUIRED** | Current spendable GFT balance. MUST NOT be negative. |
| `total_earned` | number | **REQUIRED** | Cumulative GFT earned since registration. |

## 6. Node Lifecycle Protocol

### 6.1 Registration

**Endpoint:** `POST /nodes/register`  
**Request body:** `NodeRegistration`  
**Response:** `NodeClassification`  
**Status codes:** 201 Created, 409 Conflict (duplicate public key)

Upon receiving a registration request, the coordinator MUST:

1. Verify the `public_key` is not already registered. If it is, return 409.
2. Compute the AI Performance Index from benchmark scores: `API = 0.5 × matmul + 0.3 × memory + 0.2 × latency`
3. Assign a hardware class and multiplier using the following table:

| Hardware Class | API Score Range | Multiplier |
|---|---|---|
| `mobile_edge` | 0.00 – 0.04 | 0.05× |
| `cpu_only` | 0.05 – 0.14 | 0.10× |
| `entry_consumer_gpu` | 0.15 – 0.34 | 0.50× |
| `mid_consumer_gpu` | 0.35 – 0.54 | 1.00× |
| `high_consumer_gpu` | 0.55 – 0.69 | 2.00× |
| `prosumer_gpu` | 0.70 – 0.81 | 3.00× |
| `professional_accel` | 0.82 – 0.91 | 5.00× |
| `datacenter_accel` | 0.92 – 1.00 | 8.00× |

Multipliers MUST be linearly interpolated between tier boundaries to prevent cliff effects. A node whose API score falls between two tier boundaries receives an interpolated multiplier value.

4. Set initial `reliability_factor` to `1.0`.
5. Set initial `status` to `"active"`.
6. Record the node in the registry.
7. Return `NodeClassification`.

---

### 6.2 Heartbeat

**Endpoint:** `POST /nodes/{node_id}/heartbeat`  
**Response:** `{ "status": string, "last_seen": ISO 8601 }`  
**Status codes:** 200 OK, 404 Not Found

Nodes MUST send a heartbeat at a coordinator-defined interval. The RECOMMENDED interval is 30 seconds. Nodes that miss heartbeats for longer than the coordinator's offline timeout MUST be marked offline and their shards removed from pipeline discovery.

The RECOMMENDED offline timeout is 120 seconds (four missed 30-second heartbeats).

If a node sends a heartbeat and the coordinator returns 404, the node MUST re-register its shards. This handles coordinator restarts gracefully.

---

### 6.3 Hardware Update

**Endpoint:** `POST /nodes/{node_id}/hardware-update`  
**Request body:** `NodeRegistration` (new hardware profile)  
**Response:** `{ "new_multiplier": number, "quarantine": object, "status": string }`  
**Status codes:** 200 OK, 404 Not Found

When a node's hardware changes, it MUST submit a hardware update. The coordinator MUST:

1. Compute the new multiplier from the updated benchmark scores.
2. If the new multiplier exceeds the old multiplier, impose a quarantine period:

```
Q_duration = Q_base × log2(M_new / M_old)
```

Where `Q_base` is the coordinator's configured base quarantine duration (RECOMMENDED: 24 hours) and `M_new`, `M_old` are the new and old multipliers respectively.

3. During quarantine, the node MAY continue serving inference but MUST NOT earn tokens.
4. Set `quarantine_until` to `now + Q_duration`.
5. Hardware downgrades (new multiplier ≤ old multiplier) do NOT trigger quarantine.

---

### 6.4 Deregistration

**Endpoint:** `DELETE /nodes/{node_id}`  
**Response:** `{ "status": "deregistered" }`  
**Status codes:** 200 OK, 404 Not Found

A node MAY deregister gracefully. The coordinator MUST remove the node's shards from the pipeline registry upon deregistration. Token balances are retained and remain redeemable after deregistration.

---

### 6.5 Node Status Model

| Status | Description |
|---|---|
| `active` | Node is online, registered, and eligible to earn tokens |
| `quarantine` | Node has submitted a hardware upgrade and is in the quarantine period |
| `suspended` | Node has failed enough reliability challenges to trigger suspension |
| `offline` | Node has missed heartbeats beyond the offline timeout |
| `pending` | Node has registered but benchmark scores are not yet submitted |

## 7. Shard Protocol

### 7.1 Shard Registration

**Endpoint:** `POST /inference/shards/register`  
**Request body:** `ShardRegistration`  
**Response:** `{ "shard_id": string, "node_id": string, "model_id": string, "layers": string, "layer_count": integer }`  
**Status codes:** 200 OK, 400 Bad Request (invalid layer range or unknown model)

The coordinator MUST validate:

- The `model_id` is a recognized model in the coordinator's model registry
- `layer_start` >= 0 and `layer_end` < total model layers
- `layer_start` <= `layer_end`
- The `transports` array contains at least one entry

A node MAY register multiple shards for different models. A node MUST NOT register overlapping layer ranges for the same model.

---

### 7.2 Shard Heartbeat

**Endpoint:** `POST /inference/shards/heartbeat/{node_id}`  
**Response:** `{ "status": "ok", "node_id": string, "shards": integer }`  
**Status codes:** 200 OK, 404 Not Found (node not in shard registry)

Shard heartbeats are separate from node heartbeats. Nodes MUST send shard heartbeats at the same interval as node heartbeats. If the shard heartbeat returns 404, the node MUST re-register its shards. This is the mechanism by which nodes detect coordinator restarts and re-establish their shard registrations without manual intervention.

---

### 7.3 Shard Deregistration

**Endpoint:** `DELETE /inference/shards/{node_id}`  
**Query param:** `model_id` (optional — if absent, removes all shards for the node)  
**Response:** `{ "status": "deregistered", "shards_removed": integer }`  
**Status codes:** 200 OK, 404 Not Found

---

### 7.4 Layer Coverage Rules

For a shard registration to be valid, the declared layer range MUST satisfy:

- `layer_start` >= 0
- `layer_end` < `model.total_layers`
- `layer_start` <= `layer_end`

For a pipeline to be complete, the union of all shard layer ranges MUST satisfy:

- Coverage begins at layer 0
- Coverage ends at `model.total_layers - 1`
- No gaps exist between consecutive shards (shard N's `layer_end + 1` == shard N+1's `layer_start`)

Overlapping layer ranges are permitted in the registry (multiple nodes may serve the same layers). The pipeline scheduler selects a non-overlapping cover.

---

## 8. Pipeline Protocol

### 8.1 Pipeline Discovery

**Endpoint:** `GET /inference/pipeline/{model_id}`  
**Response:** `Pipeline`  
**Status codes:** 200 OK, 404 Not Found (model not recognized)

Returns the current best pipeline for the requested model. The response includes `is_complete` to indicate whether the pipeline can serve inference. An incomplete pipeline MUST NOT be used for inference — the coordinator MUST route to fallback.

---

### 8.2 Pipeline Completeness

A pipeline is complete if and only if:

1. At least one online shard covers layer 0
2. At least one online shard covers layer `total_layers - 1`
3. For every layer L in [0, total_layers-1], at least one online shard covers L
4. A non-overlapping ordered sequence of shards can be selected that covers all layers

The coordinator MUST verify pipeline completeness before returning `is_complete: true`.

---

### 8.3 Pipeline Selection

When multiple valid complete pipelines exist (e.g. redundant shards), the coordinator SHOULD select the pipeline that optimizes for:

1. **Reliability** — prefer nodes with higher reliability factors
2. **Grid efficiency** — prefer nodes in regions with lower current grid load (grid-aware scheduling)
3. **Latency** — prefer nodes geographically close to each other to minimize activation passing latency

Pipeline selection policy is implementation-defined. The coordinator MUST document its selection policy.

---

## 9. Activation Protocol

### 9.1 Transport Declarations

Every shard node declares its supported activation passing transports in its `ShardRegistration`. The `transports` array MUST contain at least one entry. The RECOMMENDED minimum is support for the HTTP/JSON reference transport defined in §9.5.

Nodes that support higher-performance transports (gRPC, WebSocket) SHOULD declare them with lower `priority` values so the pipeline executor prefers them when both nodes in a consecutive pair support them.

---

### 9.2 Transport Negotiation

When the pipeline executor constructs a pipeline, it selects the transport for each consecutive shard pair as follows:

1. Find the intersection of transports supported by shard N and shard N+1.
2. Select the transport with the lowest combined `priority` sum from the intersection.
3. If no common transport exists, the pipeline MUST fall back to HTTP/JSON. If neither shard supports HTTP/JSON, the pipeline is invalid and the coordinator MUST NOT use it.

Transport negotiation is performed by the pipeline executor at the coordination layer, not by the shard nodes themselves. Shard nodes MUST accept connections on all declared transports simultaneously.

---

### 9.3 Tokenize

The tokenize operation converts a text prompt into input token IDs. It MUST be performed by the first shard node in the pipeline — the node serving layer 0.

**Request schema:** `TokenizeRequest` (§5.8)  
**Response schema:** `TokenizeResponse` (§5.9)

The tokenizer used MUST correspond to the `model_id`. Shard nodes MUST load and cache the tokenizer for their declared model on startup.

---

### 9.4 Forward Pass

The forward pass executes a node's assigned transformer layers on the provided input and returns activations or output tokens.

**Request schema:** `ForwardRequest` (§5.10)  
**Response schema:** `ForwardResponse` (§5.11)

**Activation format:** The format of the `activations` field is intentionally not specified at the protocol level. Activations are produced and consumed by nodes serving the same model. As long as consecutive shard nodes serve the same model, the activation format is determined by the model implementation. Implementations MUST document the activation format they produce for each supported model.

**FLOPs measurement:** The `flops_delivered` field in `ForwardResponse` MUST be measured, not estimated. Implementations SHOULD use hardware performance counters or model-level FLOPs accounting. The coordinator uses this value for token issuance — inaccurate reporting constitutes a violation of the protocol.

---

### 9.5 Reference Transport — HTTP/JSON

Every conforming activation server MUST support HTTP/JSON as a baseline transport. This ensures any two conforming nodes can always communicate regardless of what other transports they support.

**Endpoints:**

| Endpoint | Method | Request | Response |
|---|---|---|---|
| `/shard/tokenize` | POST | `TokenizeRequest` | `TokenizeResponse` |
| `/shard/forward` | POST | `ForwardRequest` | `ForwardResponse` |
| `/health` | GET | — | `{ "status": string, "model_id": string, "layer_start": integer, "layer_end": integer, "node_id": string }` |

All requests and responses MUST be valid JSON encoded as UTF-8. The `Content-Type` header MUST be `application/json`.

**Error responses:** On failure, activation servers MUST return an appropriate HTTP status code (4xx for client errors, 5xx for server errors) and a JSON body: `{ "error": string, "code": string }`.

**Timeout:** Pipeline executors MUST enforce a per-node timeout on forward pass requests. The RECOMMENDED timeout is 30 seconds. Nodes that exceed this timeout MUST be marked as failed for this pipeline execution and the coordinator MUST route to fallback.

---

## 10. Reliability Protocol

### 10.1 Challenge Dispatch

**Endpoint:** `POST /tasks/dispatch/{node_id}`  
**Response:** `ChallengeDispatch`  
**Status codes:** 200 OK, 400 Bad Request (node has no registered shards), 403 Forbidden (node not active), 404 Not Found

The coordinator dispatches periodic inference challenges to registered nodes to verify they are serving correct activations. Challenges use known prompts with pre-computed expected activation hashes maintained by the coordinator.

**Challenge rate:** The fraction of serving periods that trigger a challenge is implementation-defined. The RECOMMENDED rates are:

- New nodes (fewer than 20 completed tasks): 50% challenge rate
- Trusted nodes (reliability factor ≥ 0.95): 5% challenge rate
- All other nodes: linearly interpolated between 50% and 5% based on task count

---

### 10.2 Challenge Execution

Upon receiving a `ChallengeDispatch`, the node MUST:

1. Verify `task_id` has not been previously processed (replay protection).
2. Verify `expires_at` has not passed.
3. Run the challenge prompt through its activation server using the specified layer range and temperature 0.0 (deterministic).
4. Compute the SHA-256 hash of the resulting activations or output tokens.
5. Submit a `ChallengeSubmission` to the coordinator before `expires_at`.

Nodes MUST NOT submit results after `expires_at`. The coordinator MUST reject late submissions with status `"expired"`.

---

### 10.3 Challenge Verification

**Endpoint:** `POST /tasks/result`  
**Request body:** `ChallengeSubmission`  
**Response:** `ChallengeVerdict`  
**Status codes:** 200 OK, 404 Not Found (task not found), 409 Conflict (task already completed), 410 Gone (task expired)

The coordinator verifies the submitted `result_hash` against the pre-computed expected answer. The expected answer is generated server-side by the coordinator running the same prompt against a reference model implementation. This is the authoritative answer and is never disclosed to nodes before verification.

---

### 10.4 Reliability Factor

The reliability factor `R` is updated after each challenge:

```
R = 0.5 + 0.5 × (C_completed / C_assigned × V_consistency)
```

Where:
- `C_completed` is the number of tasks successfully completed in the trailing evaluation window
- `C_assigned` is the total tasks assigned in the trailing evaluation window
- `V_consistency` is a result consistency score in [0, 1]

`V_consistency` is `1.0` when the submitted hash matches the expected answer, `0.0` when it does not.

`R` MUST be clamped to [0.5, 1.0]. The minimum reliability factor is 0.5 — a node always earns at least half the tokens its FLOPs would otherwise warrant.

---

### 10.5 Suspension and Reinstatement

A node MUST be suspended when:

- It has completed at least 10 tasks (minimum sample size), AND
- Its reliability factor falls below 0.52

A suspended node MUST NOT receive new challenge dispatches, MUST NOT earn tokens, and MUST NOT be included in pipeline discovery.

**Reinstatement:** `POST /nodes/{node_id}/unsuspend`

Reinstatement resets `reliability_factor` to `1.0`, `tasks_assigned` to 0, and `tasks_completed` to 0. The node is returned to `"active"` status. Reinstatement policy (who may initiate, under what conditions) is implementation-defined.

---

## 11. Token Protocol

### 11.1 Token Issuance

Tokens are issued to a node upon verified completion of a serving event (real inference via the distributed pipeline). The issuance formula is:

```
T_earned = F_delivered × M_hw × R_reliability
```

Where:
- `T_earned` — GigaFLOP-Tokens credited to the node's wallet
- `F_delivered` — verified GFLOPs delivered during the serving event
- `M_hw` — the node's hardware efficiency multiplier (from §6.1)
- `R_reliability` — the node's current reliability factor (from §10.4)

**Issuance timing:** Tokens MUST be issued after the inference request completes and the ledger entry is confirmed. Pending or estimated issuances MUST NOT be credited to node balances.

**Endpoint:** Issuance is triggered internally by the coordinator after a successful distributed pipeline inference. It is not a client-facing endpoint. The resulting `LedgerEntry` with `event_type: "issuance"` is the authoritative record.

---

### 11.2 Token Redemption

Tokens are deducted from a node's balance when it submits an inference request. The redemption formula is:

```
T_spent = F_model × N_tokens × Q_precision
```

Where:
- `T_spent` — GFT deducted from the requester's balance
- `F_model` — per-token FLOP cost of the requested model (coordinator-defined per model)
- `N_tokens` — number of output tokens generated
- `Q_precision` — precision scaling factor: 1.0 for fp32, 0.5 for fp16, 0.25 for int8

**Endpoint:** `POST /tokens/redeem`  
**Balance check:** The coordinator MUST verify the requester has sufficient balance before beginning inference. If balance is insufficient, the coordinator MUST return 402 Payment Required.

**Token balance endpoint:** `GET /tokens/{node_id}/balance` → `TokenBalance`

---

### 11.3 Ledger Integrity

The token ledger is an append-only, hash-chained log. Every entry includes the hash of the immediately preceding entry, forming a tamper-evident chain.

**Chain verification:** `GET /tokens/ledger/verify` → `{ "chain_valid": boolean, "total_entries": integer, "total_supply": number }`

Coordinators MUST verify chain integrity on startup and SHOULD run periodic background verification. Any hash mismatch MUST be treated as a critical error and surfaced to operators immediately.

**Total supply invariant:** At any point in time, `total_supply = sum of all positive ledger amounts`. The total supply MUST equal the total verified FLOPs contributed to the network, normalized by hardware multipliers and reliability factors. Coordinators SHOULD publish periodic supply snapshots for external audit.

---

### 11.4 Token Expiry

Tokens earned through issuance MUST have an expiry date. The RECOMMENDED expiry period is one year from issuance. Coordinators MUST NOT expire tokens without a minimum notice period of 90 days. The RECOMMENDED mechanism is an `"expiry"` ledger event debiting the expired amount at the expiry date.

Tokens are non-transferable. A token balance is associated exclusively with the node wallet that earned it and MUST NOT be assigned, sold, or exchanged with other nodes.

---

## 12. Fallback Protocol

### 12.1 Routing Decision

The coordinator MUST route an inference request to the distributed pipeline if and only if a complete pipeline exists for the requested model. Otherwise it MUST route to the fallback service.

```
backend =
  "distributed"  if pipeline.is_complete == true
  "fallback"     otherwise
```

The fallback backend MUST always be available. Coordinators MUST NOT return an error to the client solely because the distributed pipeline is unavailable.

**Token cost differential:** Coordinators MAY charge different GFT costs for distributed vs. fallback inference. The RECOMMENDED policy is to charge higher GFT for fallback requests (reflecting the absence of volunteer subsidy) to incentivize volunteer network growth.

---

### 12.2 Fallback Interface Contract

The fallback service MUST implement a minimal interface that the coordinator can call when the distributed pipeline is unavailable. This interface is internal to the coordinator — it is not exposed to clients.

**Fallback request:**

```json
{
  "request_id": "<UUID>",
  "model_id": "<string>",
  "prompt": "<string>",
  "max_new_tokens": "<integer>",
  "temperature": "<number>",
  "precision": "<enum>"
}
```

**Fallback response:**

```json
{
  "text": "<string>",
  "output_tokens": "<integer>",
  "input_tokens": "<integer>",
  "latency_ms": "<number>"
}
```

The fallback service implementation is not standardized — it MAY be a self-hosted model, a commercial inference API, or any other backend. The coordinator is responsible for translating between the OCIP-0001 inference request and the fallback service's native interface.

## 14. Conformance

### Node Conformance

A conforming contributor node MUST:

- Implement the node lifecycle protocol (§6): registration, heartbeat, hardware update, deregistration
- Implement the shard protocol (§7): shard registration, shard heartbeat, shard deregistration
- Support the HTTP/JSON reference transport for activation passing (§9.5)
- Respond to reliability challenges within the challenge expiry window (§10.2)
- Submit accurate `flops_delivered` measurements in `ForwardResponse` (§9.4)
- NOT submit challenge results after `expires_at`
- NOT claim tokens for FLOPs not actually delivered

A conforming contributor node SHOULD:

- Support at least one additional transport beyond HTTP/JSON
- Declare `ocip4_node_id` if participating in OCIP-0002 governance

### Coordinator Conformance

A conforming coordinator MUST:

- Implement all endpoints defined in §13
- Maintain an append-only, hash-chained token ledger (§11.3)
- Enforce the token issuance formula (§11.1)
- Enforce the token redemption formula (§11.2)
- Enforce pipeline completeness before routing to distributed backend (§8.2)
- Always route to fallback when no complete pipeline exists (§12.1)
- Implement the reliability challenge protocol (§10)
- Enforce challenge quotas and expiry
- Apply hardware quarantine on multiplier upgrades (§6.3)
- Verify ledger chain integrity on startup

A conforming coordinator SHOULD:

- Implement grid-aware pipeline selection (§8.3)
- Publish periodic total supply snapshots for external audit
- Document hardware multiplier values, reliability thresholds, challenge rates, and token expiry periods

### Full Pipeline Conformance

A conforming full pipeline implementation MUST satisfy both node and coordinator conformance and additionally MUST:

- Implement transport negotiation between consecutive shard pairs (§9.2)
- Propagate `request_id` through all shard calls
- Return `ledger_ref` and `confirmed_at` in `InferenceResponse` for OCIP-0003 integration
- Handle node failure mid-pipeline by routing to fallback

## 15. Security Considerations

### Compute Fraud

**Result fabrication:** A node may return plausible-looking challenge results without executing the computation. Mitigated by the reliability challenge protocol — challenges use known prompts with pre-computed expected hashes that cannot be guessed without genuine execution.

**FLOPs inflation:** A node may report higher `flops_delivered` than actually computed. Mitigated by coordinator-side FLOPs estimation cross-checking. Coordinators SHOULD maintain per-model, per-layer FLOPs estimates and flag submissions that deviate significantly from expectations.

**Replay attacks:** A node may attempt to resubmit a previously accepted challenge result. Mitigated by the `task_id` deduplication requirement — coordinators MUST reject duplicate `task_id` submissions.

### Identity Attacks

**Sybil attacks:** A single operator registers multiple node identities to multiply token earnings. Mitigated by hardware fingerprinting in `HardwareProfile.fingerprint`. Coordinators SHOULD consolidate nodes sharing a hardware fingerprint under a single identity.

**Hardware swap attacks:** A node registers low-end hardware, earns baseline tokens, then upgrades to high-end hardware without the quarantine penalty. Mitigated by the quarantine formula in §6.3.

### Token Attacks

**Token forgery:** An attacker attempts to insert false ledger entries. Mitigated by the hash-chained ledger — any inserted or modified entry breaks the chain and is detectable by chain verification.

**Balance theft:** An attacker attempts to deduct tokens from another node's balance. Mitigated by request authentication — all redemption requests MUST be authenticated by the requesting node's private key.

### Network Attacks

**Pipeline poisoning:** A malicious node participates in a pipeline and returns corrupted activations. Mitigated by the reliability challenge protocol — nodes serving incorrect activations are detected and suspended. For high-value inference, coordinators MAY implement redundant pipeline execution.

**Coordinator impersonation:** A malicious actor runs a fake coordinator to steal node registrations. Mitigated by nodes verifying coordinator identity via TLS certificate or public key pinning. Implementations SHOULD document their coordinator authentication mechanism.

## 16. Versioning

OCIP-0001 follows the same versioning policy as OCIP-0005. See OCIP-0005 §10.

The activation protocol has its own version dimension via the `protocol_version` field in `NodeRegistration`. Changes to the coordination API that are not backwards compatible MUST result in a MAJOR version increment of OCIP-0001. New optional endpoints or fields are MINOR changes.

## 17. Relationship to OCIP-0002 through OCIP-0005

| Spec | Relationship |
|---|---|
| **OCIP-0005** | Conversation portability layer. Independent of OCIP-0001. A conversation exported via OCIP-0005 may reference inference served by an OCIP-0001 network but does not depend on it. |
| **OCIP-0004** | Project layer. Independent of OCIP-0001. |
| **OCIP-0003** | Group session layer. Directly integrated. The `InferenceResponse.ledger_ref` and `confirmed_at` fields map directly to `SessionTurn.inference_cost.ledger_ref` and `confirmed_at` in OCIP-0003. The `InferenceRequest.session_id` and `participant_id` fields enable per-turn cost attribution in OCIP-0003 sessions. |

Nodes that wish to participate in governed OCIP-0003 sessions SHOULD have both an OCIP-0001 node identity and an OCIP-0002 node identity linked via `ocip4_node_id`. |

## 18. Examples

### Full Node Registration Flow
